# 最终交付前未完成事项清单

这份清单用于回答“当前项目离最终交付给用户正式验收，还差什么”。
这里的“最终交付”不是指本地 Docker + sandbox 页面验收，而是指面向真实用户或真实联调对象的正式交付前状态。

## 当前已完成的基线

当前已经完成：

- `P0-P4` 既定计划内代码实现
- 本地 Docker 环境联通
- 本地 stateful sandbox downstream contract
- 页面化验收工作台
- 单项异常注入、补偿、重试、查询、退款、审计等本地验收

因此，下面列的是**真正还没完成、且影响正式交付**的事项。

## 最新进展（2026-04-24）

今天已经前移的事项：

- 真实外部下游契约接入的仓库内切换能力已补齐：
  - 本地 sandbox provider / store 已支持 `gateway.downstream.sandbox.enabled`
  - Dubbo consumer 已不再强制 `injvm=true`
  - 当前剩下的是“真实外部 provider 契约本身”和真实联调环境，不再是仓库内硬绑定问题
- 商户安全配置已前移一部分：
  - 验签链路已通过 `MerchantCredentialProvider` 与具体配置结构解耦
  - `gateway-security.json` 已进入 local / docker profile 的 Nacos 导入链路
  - 商户配置 bean 已具备动态刷新边界
- 交付级文档已补齐一套基础材料，见 `docs/delivery/`
- 本地组件故障探针已固化为脚本：
  - `scripts/local-reliability-suite.ps1`
  - 已完成一轮真实 Docker 验证，证据目录为 `.tmp-reliability/20260424-113208`
  - 当前本地结果：
    - `redis-outage => PASS`
    - `mysql-outage => PASS`
    - `rocketmq-broker-outage => PASS`
    - `seata-outage => REVIEW`

因此，本清单中的未完成项已经进一步收敛为“真实外部联调物料与最终架构决策”。

## P0：必须完成

### 1. 真实外部下游契约接入

当前状态：

- 支付/退款下游默认仍然是本地 sandbox contract
- 但仓库内已经补齐外部切换能力，不再强依赖本地 `injvm` provider

未完成内容：

- 替换本地 Mock Dubbo provider
- 接入真实外部支付创建契约
- 接入真实外部支付查询契约
- 接入真实外部退款创建/查询契约
- 收口真实下游错误码、超时、拒绝、处理中语义

为什么是 P0：

- 这是从“技术可验”进入“真实可交付”的分水岭
- 不完成这一项，当前验收只能证明本地脚手架和本地契约成立

### 2. 真实商户配置与密钥管理方案落地

当前状态：

- 当前已补齐统一商户凭据 provider、Nacos 导入链路和动态刷新边界
- 但真实密钥托管与最终配置源策略还未定板

未完成内容：

- 商户密钥最终存放方案定板
- 接入配置最终来源定板：`Nacos / MySQL / 混合`
- 明确密钥轮换、禁用、失效和灰度切换方式

为什么是 P0：

- 真正交付前，商户接入物料和签名体系必须是正式方案

### 3. 正式端到端验收基线

当前状态：

- 本地页面验收已具备，但是真实交付口径下还不够

未完成内容：

- 真实下游接入后的完整端到端验收脚本
- 正式验收所需请求样例、验签样例、错误码样例
- 至少一轮真实联调环境证据沉淀

为什么是 P0：

- 用户正式验收不能只依赖本地 sandbox 结果

## P1：强烈建议在正式验收前完成

### 4. Redis / MySQL / RocketMQ / Seata 组合故障注入验收

当前状态：

- 单点异常和局部恢复链路已具备
- 已新增本地可靠性脚本并完成一轮真实 Docker 探针

未完成内容：

- Redis 故障 + 幂等/锁语义联动验证
- MySQL 故障 + 事务/日志/状态边界验证
- RocketMQ 故障 + Outbox/消费/重试/死信联动验证
- Seata 故障 + 事务提交/回滚边界验证
- 组合场景自动化脚本或明确操作手册

为什么是 P1：

- 这是正式交付前验证系统“不会在局部故障下变成黑洞”的关键证据
- 目前“单组件本地故障探针”已经有了，但“跨组件组合故障 + 正式联调口径结论”还没收口

### 5. 支付最终状态策略收口

当前状态：

- 当前闭环已能工作，但“回调为主”还是“查询兜底 + 回调纠偏”尚未最终定板

未完成内容：

- 明确最终状态源
- 明确状态推进优先级
- 明确回调缺失、查询超时、查询和回调冲突时的最终规则

为什么是 P1：

- 这是用户最关心的业务一致性问题之一

### 6. 退款域最终模型收口

当前状态：

- 退款创建、查询、回调已完成，但模型是否长期独立仍未最终定板

未完成内容：

- 退款表模型和聚合边界最终决定
- 退款与支付主单的关联策略确认
- 退款补偿和异常边界补充

为什么是 P1：

- 正式交付后再改退款模型，代价会明显变高
