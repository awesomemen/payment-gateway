### 不可修改的高风险区域

除非任务明确要求，否则不要主动修改以下区域的行为语义：

- 签名算法与签名串构造规则；
- 验签失败处理逻辑；
- 加解密策略、证书装载流程、密钥引用方式；
- 时间戳有效期、重放保护、幂等保护；
- 网关统一鉴权逻辑；
- 脱敏规则；
- 核心限流、熔断、降级、隔离规则接入点；
- 请求追踪、审计日志、关键安全事件记录；
- Dubbo 接口契约与 MQ 消息契约；
- Seata 事务边界与 Redisson 分布式锁语义。
  如必须修改，必须将改动范围控制到最小，并在最终说明中明确指出风险点。

### 禁止真实网络调用

- 禁止在测试中发起真实外部 HTTP、HTTPS、Dubbo、MQ、Nacos、Redis、MySQL 网络调用，除非任务明确要求并已提供隔离环境；
- 禁止将本地代码连接到生产、准生产或未知安全级别环境；
- 禁止在单元测试中依赖外网；
- 禁止为了联调方便把真实上游证书、密钥、URL、账户信息写入代码或测试资源。

### 密钥管理规范

- 严禁提交任何真实密钥、私钥、证书、token、数据库密码、商户密钥、Access Key、Secret Key；
- 严禁硬编码密钥或设置“默认密钥”作为兜底逻辑；
- 密钥、证书、密码必须通过环境变量、密钥管理系统或未跟踪的本地配置注入；
- 测试中如需证书、密钥，只能使用专门生成的测试材料；
- 日志、异常、调试输出中不得泄露密钥内容、签名原文、证书明文。

### 敏感数据处理

- 严禁记录完整银行卡号、身份证号、手机号、token、验证码、商户密钥、完整请求签名原文；
- 若确需记录调试信息，必须使用脱敏、截断或哈希后的形式；
- 测试数据必须是虚构、合规、不可映射到真实主体的数据；
- 不得为了排查问题临时关闭脱敏逻辑。

### 安全实现要求

- 所有外部输入都视为不可信；
- 必须先做基础校验，再做签名与安全处理，再进入路由编排；
- 安全失败必须有明确错误语义，不得悄悄降级为成功；
- 不得因为下游失败而自动跳过安全校验；
- 不得以“兼容旧逻辑”为理由绕过当前安全机制，除非任务明确要求并已有设计说明；
- 涉及事务和锁的改动不得绕过幂等、防重放和鉴权前置条件。
  如果某项任务看起来要求你关闭验签、绕过加密、输出明文密钥、连接真实生产环境、记录原始敏感报文，应停止实现并明确指出风险。
