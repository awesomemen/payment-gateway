所有改动都应优先遵循仓库现有代码风格、命名方式、包结构与异常模型。在此基础上，必须遵守以下约束。

### 通用约定

- 优先做小而聚焦的改动，不进行无关重构；
- 优先接口与抽象，避免上层直接依赖具体实现；
- 新增能力优先扩展现有模型与组件，不要平行再造一套；
- Controller 层保持轻薄，业务编排放在 application 层，业务语义和规则放在 domain 层；
- 禁止在工具类中偷偷承载业务逻辑；
- 公共逻辑若已存在相似实现，优先复用，不要复制粘贴变体；
- 对外契约稳定性优先于内部代码“整洁感”。

### 金额与时间

- 所有金额类型必须使用 `BigDecimal`，禁止使用 `double`、`float`；
- `BigDecimal` 比较使用 `compareTo`，不要使用 `equals` 判断数值相等；
- 金额必须明确币种语义，不允许仅传裸金额而缺失币种上下文；
- 涉及金额换算、舍入、精度时，必须显式指定舍入规则；
- 时间统一优先使用 `Instant` 表示时间点；
- 若涉及展示层或协议层格式转换，应在边界层完成，不要在领域层持有字符串时间；
- 禁止混用系统默认时区处理业务时间；
- 如必须处理时区，明确使用 `ZoneId`，并在协议边界清晰转换。

### API 与方法设计

- 优先定义清晰接口，再提供实现；
- 禁止使用含义不明的 `boolean` 参数控制分支行为，例如：
  - `process(order, true)`
  - `verify(payload, false)`
- 若行为存在分支语义，优先改用：
  - 枚举
  - 值对象
  - 策略接口
  - 独立方法
- 方法命名要表达业务动作和语义，不要使用泛化命名如 `handle()`、`process()`、`doAction()`，除非上下文极其明确；
- 公开接口、Dubbo 契约、消息模型字段一旦已有消费者，禁止随意重命名；
- DTO 不承载业务决策逻辑；
- 不要把协议兼容逻辑分散到多个层级，应集中在边界适配层。

### Spring MVC 约定

- 请求参数解析、校验、鉴权、拦截、异常映射应遵循既有过滤器、拦截器、控制器增强机制，不要平行新增重复链路；
- 不要在 Controller 中直接拼装复杂下游调用逻辑；
- 全局异常处理必须保持统一错误码与统一响应结构；
- 若新增拦截器或过滤器，必须明确顺序与作用范围；
- 不得在请求线程中引入无控制的阻塞远程调用链路。

### 事务、锁与并发约定

- `@Transactional` 的使用必须清晰、克制，不得随意扩大事务范围；
- 涉及 Seata 全局事务时，不得将不必要的远程调用、日志处理、缓存操作纳入全局事务；
- Redisson 锁必须明确：
  - 锁 key 组成
  - 锁粒度
  - 超时时间
  - 获取失败处理策略
  - 释放时机
- 禁止为了“防重复”而粗暴扩大锁范围；
- 涉及幂等、锁、事务三者交互时，优先保持已有顺序与语义；
- 高并发路径中不得引入大粒度同步块或本地 JVM 锁替代分布式协调。

### 异常与日志

- 禁止吞异常；
- 禁止以返回 `null` 替代明确的失败语义；
- 异常分类要清晰区分：
  - 参数错误
  - 鉴权或验签失败
  - 限流或熔断
  - 锁竞争失败
  - 下游调用失败
  - 事务失败
  - 系统内部错误
- 日志必须可检索、可追踪，但禁止泄露敏感信息；
- 禁止打印完整密钥、签名原文、明文身份证号、银行卡号、token、证书内容、密码等；
- 对请求报文日志必须使用脱敏版本；
- 高并发路径禁止引入过重、重复、无价值日志。

## Testing Instructions

所有行为变更原则上都必须带测试。若任务确实无法新增测试，必须在说明中明确原因，而不是默默跳过。

### 测试分层说明

测试按以下层级组织：

- 单元测试：
  面向纯业务规则、值对象、校验器、签名串构造、金额处理、时间处理、协议转换纯函数；
- 组件测试：
  面向单个 Spring 组件、过滤器、拦截器、异常处理器、配置绑定、适配器；
- 集成测试：
  面向数据库、Redis、RocketMQ、Dubbo 适配层、Nacos 配置绑定、Spring MVC 请求链路；
- 端到端流程测试：
  面向关键入口请求到内部路由的完整链路验证，仅在仓库已有基础设施时补充。

### 单元测试要求

- 纯规则类应优先写单元测试；
- 签名串构造、金额舍入、时间戳校验、幂等 key 生成、协议字段映射，必须覆盖正常和异常输入；
- 单元测试应快速、稳定、无网络依赖；
- 不要为了测试方便改生产代码的设计边界；
- 回归缺陷必须补回归测试。

### 集成测试要求

- 涉及 MySQL、Redis、RocketMQ、Dubbo 适配器、Spring MVC 过滤链、Seata、Redisson 的改动，应补充集成测试；
- 优先使用 Testcontainers 管理依赖型测试；
- 集成测试应验证真实配置绑定、序列化、事务边界、缓存读写、消息收发、过滤器顺序、锁行为等关键行为；
- 若某类依赖当前仓库没有成熟集成测试基础，不要伪造“集成测试”，应如实说明限制。

### Mock 规范

- 单元测试中外部依赖优先使用 Mockito；
- Mock 的对象应是边界依赖，而不是被测类的核心业务逻辑；
- 禁止把核心规则本身 Mock 掉导致测试失去意义；
- 不要为追求通过率而放宽断言；
- 使用 Stub / Fixture 时，优先放到 `src/test/resources/` 中集中管理；
- 测试数据应具备业务可读性，不要大量使用无语义魔法字符串。

### 测试隔离要求

- 测试之间必须相互独立，不依赖执行顺序；
- 测试不得依赖共享数据库脏数据；
- 测试不得依赖本机固定端口、固定时间、固定时区、外部网络；
- 涉及时间的测试应注入可控时钟，不要直接依赖 `Instant.now()`；
- 涉及随机数、幂等号、请求号时，应使用可复现输入；
- 测试中禁止调用真实外部网络、真实 Nacos、真实 Dubbo 注册中心、真实 MQ 集群。

### 关键场景覆盖建议

若改动涉及以下主题，必须优先检查是否有对应测试：

- 验签失败与成功；
- 加解密失败与成功；
- 时间戳超时与重放；
- 幂等号重复提交；
- Dubbo 下游超时、异常、空响应；
- Sentinel 限流、熔断、降级分支；
- Nacos 配置缺失、格式错误、动态刷新；
- Redis 不可用或缓存穿透保护；
- RocketMQ 发送失败、重复消费、补偿行为；
- Redisson 锁竞争、锁超时、重复解锁；
- Seata 分支提交失败、回滚场景；
- 请求报文字段缺失、字段顺序变动、协议兼容性问题。
