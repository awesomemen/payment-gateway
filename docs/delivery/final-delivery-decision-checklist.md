# 最终交付决策与输入清单

本文档用于把“仓库内已完成”和“正式用户验收前仍需外部输入”的边界讲清楚。

当前仓库已经具备本地 Docker 技术脚手架、页面化验收、基础业务闭环、可靠性探针和交付文档骨架。正式交付前，下列事项必须由业务方、下游系统方或部署方提供最终输入，仓库才能进入真实用户验收。

## 1. 下游支付服务契约

必须确认：

- 真实 Dubbo registry 地址、namespace、group、version。
- 支付创建、支付查询、退款创建、退款查询服务接口名和方法签名。
- 请求 DTO 与响应 DTO 字段映射表。
- 下游业务错误码、拒绝码、处理中码、超时语义映射表。
- 幂等键、商户订单号、渠道流水号、退款单号的唯一性规则。
- 下游 SLA、超时配置、重试边界和熔断策略。

交付准入标准：

- `gateway.downstream.sandbox.enabled=false` 时，网关可以连通真实 provider。
- 支付创建、支付查询、退款创建、退款查询至少各完成一组成功和一组失败证据。
- 下游返回的错误语义可以稳定映射为网关统一错误码。

## 2. 商户密钥与配置来源

必须确认：

- 商户密钥最终存放位置：Nacos、MySQL、KMS 或混合方案。
- 密钥引用方式：明文配置、密文配置、证书引用或外部密钥 ID。
- 密钥轮换流程、灰度窗口、回滚流程和禁用策略。
- 商户启停、限流、回调地址和验签算法的配置归属。
- 生产环境配置变更审批与审计要求。

交付准入标准：

- 正式商户配置不依赖仓库内示例密钥。
- 至少完成一组正式商户验签成功、签名错误、重放请求、过期请求证据。
- 密钥轮换或禁用的操作步骤可执行，并能被审计。

## 3. 支付最终状态策略

必须确认：

- 最终状态来源：回调优先、查询兜底，还是查询优先、回调纠偏。
- 回调缺失、查询超时、查询与回调冲突时的状态推进规则。
- 处理中订单的最大等待时间、补查频率和人工介入边界。
- 成功、失败、关闭、退款中的终态定义。

交付准入标准：

- 每个非终态都有明确的补偿路径。
- 不存在“HTTP 成功但订单长期无最终状态”的未定义场景。
- 验收证据能证明状态推进规则和数据库最终状态一致。

## 4. 退款域最终模型

必须确认：

- 退款是否作为独立聚合长期演进。
- 退款单与支付主单的关联、拆分退款、重复退款和部分退款规则。
- 退款失败、退款处理中、退款回调冲突的补偿策略。
- 退款金额校验、累计退款金额和并发退款边界。

交付准入标准：

- 退款创建、查询、回调、补偿都有明确数据模型和状态机。
- 多次退款或部分退款不会破坏支付主单一致性。
- 验收证据覆盖退款成功、退款失败、退款处理中和重复请求。

## 5. Seata 事务策略

必须确认：

- Seata 不可用时是否允许受理支付请求。
- 如果允许受理，哪些数据一致性风险由 Outbox、补偿任务或人工流程兜底。
- 如果不允许受理，网关应返回的错误码和 HTTP 状态。
- 全局事务超时、分支事务失败、TC 恢复后的补偿策略。

当前本地结论：

- 本地可靠性探针中 `seata-outage` 结果为 `REVIEW`。
- 该结果不是代码错误，而是事务策略尚未定板导致不能自动判定通过或失败。

交付准入标准：

- Seata 掉线场景有明确产品级结论。
- 本地可靠性脚本或正式验收脚本能按照最终策略自动判定 PASS/FAIL。

## 6. 正式验收证据包

必须收集：

- 应用版本、Git commit、镜像 tag、部署时间。
- 环境配置清单和敏感项脱敏说明。
- 健康检查、注册中心、MQ、数据库、缓存、Seata、观测平台截图或导出记录。
- 支付成功、支付失败、支付查询、支付回调证据。
- 退款成功、退款失败、退款查询、退款回调证据。
- 幂等、验签失败、重放请求、过期请求证据。
- Redis、MySQL、RocketMQ、Seata 故障注入结论。

交付准入标准：

- 每条核心验收用例都有请求、响应、数据库状态和必要的 MQ/审计证据。
- 证据包可以从 Git commit 和环境配置追溯到对应部署。
- 未通过项都有明确负责人、修复计划或验收豁免结论。

## 7. 交付判定

仓库侧已完成项：

- 本地 Docker 技术组件联通。
- 最小化技术脚手架和核心业务闭环。
- 页面化功能验收工作台。
- 单测和本地 Docker 可靠性验证。
- Redis/MySQL 基础设施不可用的稳定错误语义。
- 交付文档骨架和本地证据索引。

正式验收前必须由外部输入完成的项：

- 真实下游 provider 契约与联调环境。
- 正式商户密钥托管和配置来源决策。
- 支付最终状态策略。
- 退款域最终模型。
- Seata 不可用策略。
- 正式验收证据包。

结论：

当前项目可以交付给开发、测试或联调团队继续接入真实外部资源；在上述外部输入未完成前，不能声明为“面向最终用户的正式验收完成”。
