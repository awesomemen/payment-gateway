本节定义架构护栏，代理必须在这些约束内进行实现，避免把支付网关逐步改造成边界模糊、难以治理、难以审计的系统。凡是改动涉及接口边界、安全边界、事务边界、流量治理边界、可观测性边界，都必须优先满足本节要求，而不是优先追求“代码更少”或“实现更快”。

### 分层调用约束

必须遵守以下调用方向：

- `gateway-web` 可以依赖 `gateway-application`、`gateway-api`、`gateway-common`
- `gateway-application` 可以依赖 `gateway-domain`、`gateway-infrastructure`、`gateway-security`、`gateway-common`
- `gateway-domain` 只应依赖 `gateway-common` 与纯领域抽象，不应依赖 Web、DB、MQ、Dubbo、Nacos、Servlet API
- `gateway-infrastructure` 负责实现外部依赖接入，不应反向持有 Controller 逻辑
- `gateway-security` 提供安全能力，不应承载复杂业务编排
- `gateway-governance` 提供流量治理接入，不应侵入业务领域模型
- `gateway-observability` 提供指标、trace、审计支持，不应承载交易决策逻辑

禁止出现以下情况：

- Controller 直接操作 DAO、Mapper、Redis、MQ、Dubbo Client；
- Domain 层直接依赖 Spring MVC、ServletRequest、Dubbo、RocketMQ、Redisson、Seata；
- Infrastructure 层反向决定上层业务分支；
- 将签名、鉴权、脱敏逻辑散落在多个层重复实现；
- 将事务、锁、幂等逻辑拆散在多个模块导致顺序不可控。

### 边界对象约束

- 外部请求 DTO、内部领域对象、持久化对象、Dubbo DTO、MQ 消息体应分层清晰，不要混用同一个类跨多个边界；
- DTO 只负责数据承载与边界适配，不承载核心业务规则；
- 不得把数据库实体直接暴露给 API；
- 不得把 Dubbo 返回对象直接透传给外部 API，除非已有明确契约规定；
- 对字段兼容性敏感的对象，修改前必须先检查是否有序列化、反序列化、日志、签名串、消息消费方依赖。

### 安全与编排约束

安全处理必须前置于业务编排。对于任何进入网关的请求，处理顺序必须优先满足“身份与来源确认、报文完整性校验、基础字段校验、时间戳与重放检查、必要的解密与签名验证”，之后才能进入路由、编排、调用下游、写缓存、发消息、加锁或开启事务。不得为了复用代码而把安全检查挪到下游调用之后，也不得因为历史兼容而让未验签请求进入核心处理路径。

应用编排层负责串联安全组件、领域规则、路由策略、下游调用和稳定性治理，但不应沉淀复杂规则判断。领域规则必须下沉到 `gateway-domain`，而不是散落在 Controller、拦截器、Dubbo 适配器或 MQ 消费者中。

### 协议转换约束

外部协议与内部协议的转换应集中在明确的适配层或 application 层边界，不得在多个模块内重复拼装。任何字段转换都必须保持可追踪性和可审计性，尤其是以下字段：

- 商户号
- 请求号 / 流水号 / 幂等号
- 金额与币种
- 时间戳
- 签名相关字段
- 渠道标识 / 来源标识 / 终端标识

禁止在多个层级重复做“轻微修正”或“容错转换”，否则很容易造成签名串不一致、日志不一致、审计不一致。若必须兼容旧协议字段，应在单一位置做兼容，并附带清晰注释说明兼容期限与触发条件。

### 事务边界约束

事务边界必须集中在 application 层或明确的事务编排入口，不得让多个下层组件隐式叠加事务。涉及 Seata 时，必须明确哪些调用处于全局事务内，哪些调用必须在事务外执行。不得将以下操作无差别纳入同一个大事务：

- 日志打印
- 指标上报
- 审计补充信息写入
- 可延迟处理的消息发送
- 非关键缓存刷新
- 与主交易成功无强一致要求的辅助调用

如果当前系统已有事务编排模式，代理不得擅自把本地事务改成全局事务，也不得把全局事务改成本地事务，除非任务明确要求并给出验证方案。

### 分布式锁与幂等约束

Redisson 锁、业务幂等、防重放是三个不同维度的控制手段，不得互相替代。锁主要解决并发竞争窗口，幂等主要解决重复请求语义，重放保护主要解决安全层重复提交风险。实现时必须保持边界清晰，不得出现以下做法：

- 用分布式锁替代幂等校验；
- 用幂等记录替代签名与时间戳校验；
- 因为有 Redisson 锁就去掉重复请求检测；
- 因为有 Seata 就放松并发控制。

涉及锁的改动必须保持锁粒度稳定，避免从“按商户 + 请求号”无意放大成“按商户全局锁”，从而导致性能退化和排队放大。

### Sentinel 流量治理约束

Sentinel 资源定义必须稳定、可理解、可运维，不得随意改名。资源名、热点参数、规则来源、降级策略、fallback 入口应统一管理，不得散落在业务代码各处。新增高风险接口时，应优先复用现有治理模式，不要临时创建风格不一致的新规则。

不得把 Sentinel fallback 写成“吞掉错误然后返回业务成功”，也不得在 fallback 中绕过安全检查、跳过审计或伪造下游成功结果。降级结果必须符合既有错误语义。

### Nacos 配置约束

Nacos 配置必须通过统一的配置对象、配置前缀或配置网关读取，不得在业务代码中零散读取原始配置键。配置项命名、默认值、刷新策略、环境隔离策略必须保持一致。以下配置尤其敏感，不得随意修改默认行为：

- 验签开关与安全策略
- 限流阈值与降级阈值
- 超时配置
- 重试配置
- 事务相关配置
- MQ topic / tag / consumer group
- 锁超时时间与自动续约策略

新增配置时，必须明确回答三个问题：是否需要动态刷新、默认值是否安全、配置缺失时系统是否应拒绝启动或拒绝处理请求。

### Dubbo 接口约束

Dubbo 接口属于强兼容边界。不得随意修改以下内容：

- 方法签名
- 字段名称
- 枚举语义
- 序列化模型
- 超时与重试默认语义
- 版本路由策略

如果必须新增字段，应优先采用向后兼容方式；如果必须调整超时、重试、mock、cluster 容错策略，应明确说明是否影响幂等、是否影响下游负载、是否影响告警与监控基线。

### RocketMQ 约束

消息模型必须保持稳定，不得随意修改 topic、tag、key 组成、序列化字段、顺序语义、消费组命名。消息发送与本地事务、全局事务、补偿逻辑之间的关系必须明确。禁止以下行为：

- 在没有兼容策略的情况下修改消息体字段含义；
- 把关键业务消息降级为“尽力发送且不告警”；
- 在消费端静默吞掉异常；
- 为了避免重复消费问题而删除幂等校验。

任何消息改动都必须考虑生产者、消费者、重试、死信、补偿任务和监控告警之间的联动影响。

### 可观测性约束

Micrometer 指标、SkyWalking trace、Prometheus 标签和 ELK 日志字段属于运维与审计合同的一部分。代理不得随意重命名、删除或改语义。以下信息应尽量保持稳定透传：

- traceId / requestId
- merchantId
- channelCode
- interfaceCode
- errorCode
- degrade / block / fallback 标记
- lock / transaction / mq 关键状态

新增埋点时，标签维度应控制数量，避免高基数标签导致监控系统压力上升。不得把敏感数据放入日志、trace tag 或指标标签中。

### 异常模型约束

统一异常模型必须保持清晰。参数错误、安全错误、限流熔断、锁竞争、事务失败、下游失败、系统错误必须可区分、可统计、可追踪。不要把所有异常都包装成同一个笼统错误码，也不要为了减少分支而吞掉具体错误语义。

全局异常处理器、统一响应包装、错误码映射属于高敏感区域，除非任务明确要求，否则只允许最小范围修改。

### 数据访问约束

DAO、Mapper、Repository 只负责数据访问，不承载业务决策。不得在 SQL 层偷偷做业务状态修正，也不得在缓存读写逻辑中悄悄补业务分支。缓存应被视为性能优化层，而不是唯一真实来源，除非已有设计明确说明。涉及 Redis key 结构、TTL、序列化格式的修改，必须评估兼容性与灰度影响。

### 兼容性约束

除非任务明确要求，不得主动做以下破坏性变更：

- 修改公开 API 字段名；
- 修改错误码语义；
- 修改签名字段顺序；
- 修改默认时区或时间格式；
- 修改 MQ / Dubbo / Nacos 配置命名；
- 修改日志字段名、trace 字段名、指标标签名；
- 修改锁 key 组成规则；
- 修改事务传播语义。

如果任务必须涉及兼容性变更，必须在改动说明中明确写出影响面、回滚方案和最小验证集。
