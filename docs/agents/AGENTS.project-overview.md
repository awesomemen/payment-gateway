### 服务定位与职责边界

本项目是支付系统中的支付网关服务，定位为所有支付请求的统一接入层与安全边界层，目标是在高并发、低延迟场景下，为内部支付链路提供稳定、可控、可审计的入口能力，同时保障内部系统、敏感数据与核心交易链路的安全。

该服务的核心职责包括：

- 作为所有支付请求服务的统一入口，承接来自 C 端、B 端及其他上游系统的接入请求；
- 对请求进行基础参数合法性校验，包括但不限于必填字段、格式、金额、币种、时间戳、商户标识、幂等号等；
- 执行安全与保密控制，包括加签、验签、加密、解密、敏感字段脱敏与安全审计；
- 完成外部协议与内部协议之间的报文转换、字段映射与语义适配；
- 将请求路由、转发给内部服务接口，并对调用链进行超时控制、异常分类与失败治理；
- 在高并发场景下提供限流、熔断、降级、隔离与容错能力，保护后端交易系统；
- 为调用链提供统一的可观测性基础，包括日志、指标、追踪、请求标识传递与错误归因；
- 维护网关层的接口兼容性、版本演进约束与向后兼容策略。

本服务的职责边界同样需要被严格理解。以下能力**不属于**本服务的主要职责，除非任务明确要求，否则代理不应主动引入或扩展：

- 不直接承载核心资金账务记账逻辑；
- 不直接实现支付清结算主流程；
- 不直接持有或处理超出网关职责范围的商户业务规则；
- 不在网关层实现复杂业务编排，复杂业务应下沉至内部领域服务；
- 不在网关层保存不必要的敏感数据；
- 不绕过统一安全组件直接访问内部核心服务；
- 不为了“方便调试”而削弱验签、加密、鉴权、限流、分布式锁、事务控制或审计逻辑。

在设计与修改代码时，代理必须将本服务视为“**安全边界 + 协议边界 + 流量治理边界 + 稳定性防线**”，而不是普通的 CRUD 微服务。所有涉及签名、加密、时间戳校验、重放保护、幂等控制、协议转换、调用路由、限流熔断、分布式锁、事务边界的改动，都属于高风险变更。

### 技术栈（含版本号）

- 语言：Java 17
- 框架：
  - Spring Cloud Alibaba 2025.0.0.0（服务治理）
  - Spring Boot（微服务运行基础）
  - Spring MVC
- 数据库：MySQL 8.4
- 缓存：Redis 7.2.13
- 中间件：
  - MyBatis-Plus 3.5.15（ORM）
  - Dubbo（内部服务调用）
  - Sentinel（限流、降级与熔断保护）
  - RocketMQ 5.3.4（消息队列）
  - Nacos（配置中心与服务发现）
  - Seata（分布式事务）
  - Redisson（分布式锁）

- 开发工具包：MapStruct、Lombok
- 监控与可观测：
  - Micrometer（指标上报）
  - SkyWalking（链路追踪）
  - Prometheus + Grafana（监控大盘）
- 日志聚合：ELK
- 构建工具：Maven 3.9.9，Docker
- 测试工具：JUnit 5、Mockito、Testcontainers

建议代理默认按以下技术假设进行理解和修改：

- 使用 Java 17 语言特性时，应以可维护性和团队一致性为优先，不应为了“新特性”而引入不必要复杂度；
- Web 层默认基于 Spring MVC，请求处理、拦截器、过滤器、参数绑定、异常处理、上下文传递都按 Servlet 模型理解；
- 内部 RPC 默认通过 Dubbo 与下游服务交互，修改接口契约时必须关注兼容性；
- 流量治理默认依赖 Sentinel，所有新增高风险接口都需要考虑限流、熔断、降级、热点参数、系统保护等策略接入点；
- 动态配置默认通过 Nacos 下发，禁止将环境敏感配置、密钥、地址白名单等硬编码进代码；
- 异步链路默认基于 RocketMQ，涉及通知、补偿、回执、削峰等场景时应优先复用现有消息约定；
- 分布式事务由 Seata 管理时，必须优先保证边界清晰，不可随意扩大事务范围；
- 分布式锁默认通过 Redisson 实现，涉及锁粒度、锁超时、自动续约、幂等保护时必须保持保守；
- 可观测性默认接入 Micrometer、SkyWalking、Prometheus 与 Grafana，修改入口链路时必须关注指标与 trace 是否被破坏；
- 日志默认汇聚到 ELK，必须关注字段结构、脱敏与检索可用性；
- 构建与 CI 默认使用 Maven Wrapper 或 Maven 3.9.9；
- 测试默认优先使用 JUnit 5，Mock 使用 Mockito，依赖型集成测试优先使用 Testcontainers。

### 运行特征与质量目标

本服务默认面向以下运行特征：

- 高并发入口流量；
- 低延迟请求处理；
- 上下游协议异构；
- 强安全约束；
- 强审计与可观测性要求；
- 局部失败常态化；
- 对下游稳定性具备保护责任；
- 在幂等、防重放、限流、事务、锁竞争等方面具备严格一致性要求。

所有代码变更都应尽量满足以下质量目标：

- 优先保证安全性，再考虑便利性；
- 优先保证兼容性，再考虑重构美观；
- 优先保证交易请求可追踪、可诊断，再考虑日志简洁；
- 优先保证失败可控、可隔离，再考虑吞吐优化；
- 优先最小化改动面，避免无关重构；
- 优先保持接口、配置、指标、日志字段稳定，避免破坏运维侧依赖。

### 代理工作原则

代理在本仓库内工作时，必须遵守以下原则：

- 优先理解已有实现，再决定修改方式；
- 优先沿用现有模块边界、异常模型、返回模型、配置模型；
- 不因“代码更优雅”而主动进行大范围重构；
- 不擅自改变公开接口、协议字段、签名规则、鉴权逻辑、限流策略、锁语义、事务边界；
- 不在未完成验证前声称“已通过测试”；
- 对涉及安全、资金、协议兼容、事务一致性的改动保持极度保守。
