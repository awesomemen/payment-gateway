本节定义操作级安全检查项。代理在实施任何变更前后，都应对照本节检查风险，特别是在支付网关这种高并发、高敏感、高审计要求的系统中。若以下检查项无法确认安全，代理应优先停止、缩小改动范围或明确暴露风险，而不是继续“猜测式修改”。

### 通用检查原则

在开始修改前，先判断本次改动属于以下哪一类：

- 安全类：签名、验签、加解密、证书、时间戳、防重放、脱敏
- 协议类：请求字段、响应字段、映射逻辑、兼容逻辑
- 路由类：Dubbo 调用、服务发现、超时、重试、容错
- 治理类：限流、熔断、降级、隔离、热点参数
- 一致性类：幂等、锁、事务、消息、补偿
- 数据类：MySQL、Redis、缓存 key、TTL、序列化
- 可观测性类：日志、trace、指标、审计事件
- 配置类：Nacos、Profile、环境变量、默认值、动态刷新

如果一次改动跨越多个类别，必须按高风险类别要求执行检查，而不是按最低风险处理。

### 修改前检查

在真正写代码之前，必须至少确认以下几点：

- 是否已经定位到正确模块，而不是准备在错误层级修复问题；
- 是否已有现成组件、适配器、规则类、异常模型可复用；
- 是否涉及公开接口、Dubbo 契约、MQ 消息、配置键、日志字段、指标字段；
- 是否涉及幂等、防重放、签名、事务、锁、限流中的任一高风险点；
- 是否已经找到对应测试、样例报文、配置模板、历史实现；
- 是否能够用最小改动完成目标，而不是扩大到无关重构。

### 安全改动检查

当改动涉及签名、验签、加解密、时间戳、防重放、脱敏时，必须逐项确认：

- 签名串字段顺序是否保持不变；
- 字符编码、空值处理、字段拼接规则是否保持不变；
- 加解密算法、填充方式、证书别名、密钥引用方式是否保持不变；
- 时间戳窗口是否保持一致；
- 重放判定 key 是否保持一致；
- 安全失败时是否仍然返回既定错误码；
- 日志是否没有泄露签名原文、明文敏感字段、证书内容；
- 是否补充了成功与失败路径测试。

若无法确认以上任一项，禁止继续扩大修改。

### 协议与接口改动检查

当改动涉及 Controller 入参、出参、DTO、Dubbo DTO、MQ 消息体、序列化字段时，必须检查：

- 字段名是否变化；
- 字段是否新增、删除、改为必填、改默认值；
- 枚举值是否新增或改语义；
- JSON / XML / 表单字段顺序是否影响签名；
- 下游消费者是否依赖旧字段；
- 日志、审计、监控是否依赖该字段；
- 是否存在兼容层或灰度策略；
- 是否已有协议样例与回归测试覆盖。

尤其对支付请求号、商户号、金额、币种、签名字段，任何“微调”都必须视作高风险变更。

### Dubbo 调用改动检查

当改动涉及 Dubbo 接口、超时、重试、容错、版本、路由时，必须确认：

- 方法签名与序列化模型是否保持兼容；
- 是否改变了超时默认值；
- 是否改变了重试次数、容错模式、负载策略；
- 是否可能把非幂等接口误设为可重试；
- 是否会导致下游压力升高；
- 是否会影响 trace 透传、错误码映射、监控告警；
- 是否已检查调用失败、超时、空响应、异常响应路径。

若接口属于非幂等交易入口，必须对“重试导致重复处理”的风险做显式评估。

### Sentinel 改动检查

当改动涉及 Sentinel 资源、规则、fallback、blockHandler 时，必须确认：

- 资源名是否保持稳定；
- 规则加载来源是否明确；
- blockHandler 与 fallback 是否语义清晰；
- 降级返回是否符合既有错误码约定；
- 是否意外把系统错误当成业务成功返回；
- 是否保留审计日志、指标上报与告警语义；
- 是否检查热点参数、高并发场景和并发恢复后的行为。

禁止把 Sentinel 兜底逻辑写成“吞异常并返回成功”。

### Nacos 配置改动检查

当改动涉及配置项、配置绑定、默认值、动态刷新时，必须确认：

- dataId、group、namespace 是否正确；
- 配置前缀与绑定类是否一致；
- 默认值是否安全；
- 配置缺失时是拒绝启动还是使用保守值；
- 是否需要动态刷新；
- 动态刷新后是否会影响锁、事务、限流、路由等运行时行为；
- 是否会导致老环境无法兼容。

新增配置前要先问：没有这个配置时，系统应如何安全退化。

### Redis 改动检查

当改动涉及 Redis key、TTL、序列化、缓存用途、幂等记录、防重放记录时，必须确认：

- key 结构是否保持兼容；
- TTL 是否符合业务窗口；
- 是否存在老 key 清理与兼容问题；
- 序列化格式是否变化；
- Redis 不可用时系统如何降级；
- 是否会影响幂等、防重放、限流计数、锁竞争等关键语义；
- 是否存在缓存污染、缓存穿透、缓存雪崩风险放大。

特别注意：幂等 key、防重放 key、限流 key、分布式锁 key 不能混淆。

### Redisson 改动检查

当改动涉及分布式锁时，必须确认：

- 锁 key 是否足够精确，不会误伤无关请求；
- 锁粒度是否被无意放大；
- 锁超时时间是否与业务执行时间匹配；
- 是否依赖自动续约，若依赖则是否有线程上下文风险；
- 获取锁失败时是否有明确错误语义；
- 是否存在重复解锁、非持有线程解锁、finally 漏释放风险；
- 是否与幂等、事务、消息发送顺序冲突。

任何锁改动都要评估吞吐、延迟和死锁/假死风险。

### Seata 改动检查

当改动涉及 Seata 全局事务、本地事务、事务传播时，必须确认：

- 全局事务范围是否被扩大；
- 是否把不该纳入事务的远程调用、日志、缓存刷新纳入了事务；
- undo log / 分支事务行为是否会受影响；
- 回滚语义是否与幂等、补偿、消息发送一致；
- 异常传播是否会触发错误回滚；
- 是否存在事务内外数据可见性差异；
- 是否已有至少一条成功路径和一条失败回滚路径测试。

不要在未确认回滚语义前修改 Seata 注解或事务入口。

### RocketMQ 改动检查

当改动涉及消息发送、消费、topic、tag、消费组、消息体时，必须确认：

- topic / tag / key 是否保持兼容；
- 消息体字段是否变更；
- 发送是否需要与事务成功绑定；
- 消费是否具备幂等保障；
- 重试、死信、补偿链路是否仍然有效；
- 消费异常是否被正确暴露和记录；
- 是否会影响现有告警、监控、消费速率基线。

禁止为了“先让流程通”而静默吞掉消费异常。

### MySQL 改动检查

当改动涉及表结构、索引、唯一键、状态字段、事务 SQL 时，必须确认：

- 是否新增迁移脚本而不是修改已执行历史脚本；
- 是否影响历史数据兼容；
- 是否影响唯一性约束、幂等约束、查询性能；
- 是否改变事务内查询结果；
- 是否需要数据回填或灰度发布；
- 是否已有回滚方案；
- 是否已检查锁表、慢 SQL、索引失效风险。

支付网关中，任何和请求流水、幂等记录、审计表、异常表相关的 schema 变更都必须高度谨慎。

### 日志、指标、追踪改动检查

当改动涉及 ELK 日志、Micrometer 指标、SkyWalking 埋点、Prometheus 标签时，必须确认：

- 关键字段是否保留；
- traceId / requestId 是否持续透传；
- 是否引入高基数标签；
- 是否误把敏感数据写入日志或 tag；
- 是否影响告警规则、Dashboard 或检索语句；
- 是否仍能区分安全错误、限流错误、下游错误、事务错误、锁错误；
- 是否保留关键操作审计。

可观测性一旦被破坏，排障成本会急剧升高，因此不得把埋点视为“可有可无”。

### 提交前最小验证检查

在提交前，至少确认以下事项：

- 代码格式已通过；
- 变更模块的最小相关测试已执行；
- 关键高风险路径至少有一条成功测试和一条失败测试；
- 未引入真实地址、真实密钥、真实账号、真实证书；
- 未新增调试日志、临时开关、绕过校验逻辑；
- 改动说明已明确列出风险点与实际执行命令。

### PR 前人工复核清单

在提交 PR 前，代理应自查并在说明中体现：

- 这次改动是否改变了任何协议、配置、事务、锁、限流、安全默认行为；
- 这次改动是否会影响现有告警、监控、日志检索；
- 这次改动是否依赖环境特定配置；
- 这次改动是否需要灰度发布；
- 这次改动失败时是否容易回滚；
- 这次改动是否已经补了最关键的回归测试。

### 必须停止并显式提示风险的场景

若遇到以下情况，代理应停止继续实现，并明确指出风险或前置条件不足：

- 需要关闭验签、加密、鉴权、限流、审计才能“完成需求”；
- 需要接入真实生产环境或真实密钥才能验证；
- 无法确认 Dubbo / MQ / 配置 / 锁 / 事务改动的兼容性；
- 需要大范围重构才能完成一个局部需求；
- 缺少必要测试、样例报文、配置说明而无法安全修改；
- 需求本身会导致资金风险、安全风险或审计缺失。
